# 提示词设计经验

## 1. 文件与版本

- **主文件**：`app/agent/prompts.py`
- **当前**：System prompt v1.2 优化版 + 2026-06-12 v1.3 场景路由增补
- **抽取**：`EXTRACT_SYSTEM` / `EXTRACT_USER_TEMPLATE`（立项字段，独立链路）

变更日志写在文件头注释，重大行为变更建议在 `react_helpers.FEATURE_EVIDENCE_ROUTING` 打日期标记。

---

## 2. System prompt 结构（八段式）

| 段落 | 作用 | 经验 |
|---|---|---|
| 角色 | 投委会档案助手、中文、简洁 | 固定人设，减少闲聊漂移 |
| 业务边界 | 范围内 / 范围外 + 拒答模板 | **硬约束**，范围外必须 FINAL_ANSWER 拒答 |
| 场景路由表 | 意图 → 首选动作 | **最重要**：先分类再选工具 |
| 工具选择顺序 | 1–8 优先级说明 | 强调「按场景，不是固定流水线」 |
| 工具清单 | 8 工具 JSON 示例 | 字段名必须与 registry 严格一致 |
| 输出格式 | thought + tool + args | 严格 JSON，禁止 markdown 包裹 |
| 答案模板 | 1 句结论 + 要点 + 引用 | 控制长度，体感「丝滑」 |
| Few-shot | 9+ 示例 | **比规则更有效**；新 Case 优先补示例 |

---

## 3. 场景路由表（prompt 层）

prompt 里用表格让 LLM **先判断意图**：

```
问候/能力 → FINAL_ANSWER
明确 PRJ 编号 → get_project_business_data
简称/客户名 → find_project
跨项目统计 → query_mysql（不先 find）
材料正文/证据 → search_fulltext
术语 → network_dict_lookup
模糊 → ask_clarification
离题 → 拒答模板
```

**经验**：不要把所有问题都写成「单项目三步走」；跨项目、术语、问候是高频误路由来源。

## 4. Few-shot 关键示例（务必保留）

### 4.1 统计类 — lmz 材料份数

```
用户: "lmz项目下有几份材料?"
find_project → get_project_business_data → FINAL_ANSWER
```

要点：简称 `lmz` 由后端 token 化；命中后用 **汇总 materialCount**，不要 search 数文件。

### 4.2 证据类 — lmz 利率（2026-06-15 新增）

```
用户: "lmz项目的利率是多少?"
find_project → search_fulltext(query="利率 固定收益", projectCode=...) → FINAL_ANSWER
```

要点：明确写「利率在材料正文，用 search_fulltext」；**不要**写 get_project_business_data。

### 4.3 跨项目 — 不先 find

```
用户: "列一下所有待审议项目"
直接 query_mysql(project 表 where status=待审议)
```

### 4.4 拒答与追问

- 天气 → 拒答模板
- 「它最新进展」→ ask_clarification

## 5. 引擎提示 vs System prompt

| 层级 | 时机 | 内容 |
|---|---|---|
| System prompt | 每轮固定 | 规则 + Few-shot |
| `append_step_hints` | 每轮动态追加 | 根据**已执行 steps** 的下一步约束 |

**分工**：

- Prompt 负责「一般情况怎么选」
- 引擎提示负责「你已经 find 过了，别再 find」这类**状态相关**约束

引擎提示示例文案：

```
【引擎提示】步骤 2 已锁定项目 shtx26007 (lmz授信)。
用户问的是材料正文中的业务事实（如利率/条款）→ 下一步必须 search_fulltext(...)
get_project_business_data 只有汇总字段、不含材料利率正文。
```

**经验**：引擎提示要短、带具体 projectCode，比泛泛「请选合适工具」有效。

## 6. 答案模板与引用

要求结构：

```
{1 句话直接结论}

关键信息:
- ...

引用来源: [1] xxx  [2] yyy
```

规则：

- 只写有数据的块，不硬凑
- 材料证据必须标 `[1][2]` 对应标题
- 不确定时加置信度说明（系统另有 `confidence_badge` 映射）

拒答统一用 `DEFAULT_REJECT_ANSWER` 常量，prompt 与代码一致。

---

## 7. 改 prompt 的推荐流程

1. **先归类**：统计 / 证据 / 跨项目 / 术语 / 离题？
2. **补 Few-shot**：用真实用户原句（如 lmz 利率）
3. **看 LLM 仍错否**：若仍错 → 加 `maybe_upgrade_step` 规则，不只堆 prompt 字数
4. **加测试**：`test_prompts_v12.py`（文案关键字）、`test_react_helpers.py`（行为）
5. **远端验收**：TUI 流式看步骤是否实时、最终答案是否进 answer box

---

## 8. 常见 prompt 误区

| 误区 | 后果 | 做法 |
|---|---|---|
| 只写「优先 search_fulltext」无示例 | 利率问句仍走 get | 加示例 2b + 引擎升级 |
| 工具顺序写成固定 1→2→3→4 | 跨项目也先 find | 场景表写清例外 |
| Few-shot 用虚构 PRJ 号 | 与生产 code 不一致 | 用测试环境真实 code 形态 |
| 规则过长重复 | token 浪费、注意力分散 | 表格 + 少量高质示例 |
| 只靠 prompt 防死循环 | 重复 get/find | 引擎 `try_recover_*` |

## 9. EXTRACT 提示词（立项抽取）

独立链路，原则更简单：

- 输出严格 JSON，字段缺失 `null`
- 不编造
- 正文截断长度由调用方 `max_chars` 控制

与问答 Agent 共用 GLM 配置，但 **不要** 混用 ReAct 工具 prompt。
