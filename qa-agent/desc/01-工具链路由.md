# 工具链路由经验

## 1. 两类「材料问题」必须分开

不良资产档案里，用户问「材料」时意图不同，**不能一律走 `get_project_business_data`**。

| 问句特征 | 正则/关键词 | 正确工具链 | 错误链 |
|---|---|---|---|
| **统计类** | 材料、份数、几份、多少个 | `find_project` → `get_project_business_data` → `FINAL_ANSWER` | 重复 `find_project`；用 `search_fulltext` 数文件 |
| **证据类** | 利率、收益率、条款、合同、抵押、固定收益 | `find_project` → `search_fulltext` → `FINAL_ANSWER` | 用 `get_project_business_data`（**无材料正文利率**） |

实现位置：`react_helpers.py`

- `question_needs_material_stats()` — 统计类
- `question_needs_material_evidence()` — 证据类

**教训**：`get_project_business_data` 对齐 Java/DATABASE 汇总表，只有 `materialCount` 等字段；利率在 `material_version.parsed_text` 里，必须 `search_fulltext`。

---

## 2. 标准工具链（单项目）

```
用户问句
    │
    ├─ 项目不明确 ──► find_project（简称多变体：lmz → lmez）
    │
    ├─ 已知 projectCode ──► 跳过 find_project
    │
    ├─ 要汇总/份数 ──► get_project_business_data
    │
    ├─ 要正文证据 ──► search_fulltext(projectCode 限定)
    │
    └─ 数据够 ──► FINAL_ANSWER（带引用 [1][2]）
```

跨项目列表/待办：**不要**先 `find_project`，直接 `query_mysql`。

## 3. 引擎三层兜底（`react_helpers.py`）

仅靠 prompt，GLM 仍会选错工具或死循环。引擎在 **不替换 LLM** 的前提下做三道保险：

### 3.1 `append_step_hints` — 每轮 prompt 末尾追加

根据已有 steps，向 LLM 追加 `【引擎提示】`：

- 已 `find_project` + 统计问句 → 下一步**必须** `get_project_business_data`，**禁止**再 `find_project`
- 已 `find_project` + 证据问句 → 下一步**必须** `search_fulltext`，说明 biz 表无利率正文
- 已 `get_project_business_data` 且有 `materialCount` → **必须** `FINAL_ANSWER`
- 已 `search_fulltext` 有摘录 → **必须** `FINAL_ANSWER` 并引用 `parsedExcerpt`

### 3.2 `maybe_upgrade_step` — LLM 输出后、执行前改道

| 场景 | LLM 想调 | 引擎改为 |
|---|---|---|
| 证据问句 + 已锁定项目 | `get_project_business_data` / 再 `find_project` | `search_fulltext` |
| 证据问句 + 上一步已是 search 有结果 | 再 search / get | `FINAL_ANSWER`（`synthesize_evidence_answer`） |
| 统计问句 + 已有 materialCount | 再 `get_project_business_data` | `FINAL_ANSWER` |
| 统计问句 + 已 find 命中 | 再 `find_project` | `get_project_business_data` |

特性版本标记：

- `FEATURE_EVIDENCE_ROUTING = "2026-06-15"`（`/health` → `features.evidence_routing`）
- `FEATURE_DEBT_TARGET_ROUTING` / `FEATURE_COLLATERAL_*` — 债权抵押物，见 [08-债权抵押物问答.md](./08-债权抵押物问答.md)

### 3.3 `try_recover_*_loop` — 连续两步相同工具+相同参数

- `try_recover_material_count_loop`：重复 get/find → 从已有 observation 合成份数答案
- `try_recover_evidence_loop`：重复 get → 补调 `search_fulltext`；重复 search → 从摘录合成利率/抵押物答案

## 4b. 债权抵押物检索（证据类子场景）

**最常见业务问法**：债权标的 → 抵押物 → 还剩哪些/初始估值。

| 问句 | `evidence_search_query` 要点 | 合成 |
|---|---|---|
| 债权标的是什么 | 目标债权、岭兜、远期回购 | `extract_debt_target_from_hits` |
| 抵押物是什么 | 岭兜、建材二厂、抵押物、土地、厂房、设备 | `extract_collateral_from_texts` |
| 还剩哪些/初始估值 | 上述 + 估值、抵押金额、他项权证、贷款本金 | `extract_collateral_items_from_texts` → 清单 |

多轮「这个债权」依赖 `memory.last_debt_target`；单轮「xx债权下的…」由 `extract_debt_anchor_from_question` 解析。

完整经验：[08-债权抵押物问答.md](./08-债权抵押物问答.md)

## 5. `search_fulltext` 检索策略

1. **主路径**：MySQL `MATCH ... AGAINST` FULLTEXT，`parsedExcerpt` 取前 8000 字
2. **兜底**：FULLTEXT 无命中时，按 `projectCode` + 关键词 `LIKE`（最多 3 个词）
3. **摘录窗口**：命中关键词前后约 400 字，便于 LLM/引擎抽「固定收益15%」

利率抽取正则（引擎侧 `extract_rate_from_texts`）：

- `固定收益\s*(\d+(?:\.\d+)?)\s*%`
- `利率[为是：:\s]*(\d+(?:\.\d+)?)\s*%`
- `年化(?:收益|利率)?\s*(\d+(?:\.\d+)?)\s*%`

## 6. 路由决策表（与 prompts 场景路由对齐）

| 用户意图 | 首选工具 | 备注 |
|---|---|---|
| 问候 / 能力说明 | `FINAL_ANSWER` | 不调工具 |
| 项目简称/客户名 | `find_project` | 多变体一次工具内试完 |
| 已知 PRJ-编号 | `get_project_business_data` | 不 find |
| 材料份数 | find → get → answer | 引擎强制 |
| 利率/条款/抵押/债权标的 | find → search → answer | 引擎强制；抵押物须带债权锚点，见 [08](./08-债权抵押物问答.md) |
| 债权抵押物细问（还剩/估值） | find → search（加长 query）→ 清单式 FINAL | `FEATURE_COLLATERAL_DETAIL_ROUTING` |
| 跨项目待办/列表 | `query_mysql` | 不先 find |
| 术语 | `network_dict_lookup` | 可结合 search |
| 模糊指代「它」 | `ask_clarification` | 优于瞎猜 |
| 离题 | `FINAL_ANSWER` 拒答模板 | |
