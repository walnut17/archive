# 投委会档案管理系统 — 需求兼容性分析报告

> 撰写人：Sisyphus | 日期：2026-06-10 | 版本：v1.0

---

## 0. 摘要

本报告对投委会档案管理系统的**业务需求**与**现有架构**之间的兼容性进行系统评估。

评估维度：① 可兼容（现有架构直接支持）② 需扩展（需中等程度的架构演进）③ 超额设计（超出硬件/环境能力）。

**核心结论**：
- **可兼容**：~65% 的 P0 需求已在当前架构覆盖
- **需扩展**：~25% 的 P0 需求（关键事实档案、跨项目聚合、项目创建重设计）需要中等程度的架构扩展
- **超额设计**：~10% 的需求（多供应商 LLM 动态切换、流式 SSE、离线 LLM）在当前单机内网环境下不现实
- 当前硬件（4 核 16 GB）可支撑 v1.1 全部计划需求，但需要在智谱 API 可达的前提下

| 需求 | 等级 | 分析 |

| 审计日志 | ✅ 兼容 | audit_log 表 + GlobalExceptionHandler 已实现 |
| JWT 8h 过期 | ✅ 兼容 | JwtUtil.expiration = 8h |
| BCrypt 密码 | ✅ 兼容 | Spring Security BCryptPasswordEncoder |
| 白名单 SQL | ✅ 兼容 | QueryMysqlTool 白名单机制 |
| 单机部署 WinSW | ✅ 兼容 | 完整部署方案 |
| MySQL FULLTEXT ngram | ✅ 兼容 | 3 张表已建 FULLTEXT 索引 |

**关于流式输出**：需求中有 SSE 要求（Agent 思考过程逐步显示），但当前架构中：
- AgentEngine.run() 是同步阻塞的 `String callLlm(prompt)`，非流式
- 前端 Axios 120s 超时，不支持 EventStream
- 改 SSE 涉及：AgentEngine→后端 Controller SSE→前端 fetch EventSource 三层改动
- **建议降级**：等全部步骤完成一次性输出，不做流式

## 4. 超额设计分析

以下需求在当前环境/架构下被判断为超额设计：

| # | 需求 | 来源 | 原因 |
|---|------|------|------|

| 2 | 多供应商 LLM 动态切换 | ARCHITECTURE-v2.md | LLMProviderFactory 框架已搭但仅 GLMProvider 实现，单机场景下动态切换无实际收益 |

| 6 | 微服务化 | 已排除 | 单机单体架构正确，拆微服务会显著增加运维负担 |

## 5. 环境承受能力评估

### 5.1 硬件承载分析

| 负载场景 | CPU | 内存 | 磁盘 | 结论 |
|---------|-----|------|------|------|
| 日常使用（5 人并发） | < 10% | ~2 GB | - | ✅ 充裕 |
| Agent 模式高峰（5 人同时问） | ~30% | ~3 GB | - | ✅ 可承受 |
| 文件批量上传 + Tika 解析 | ~50% | ~4 GB | 大文件场景 | 🟡 Tika 是 CPU 密集型 |
| 大量全文检索 | ~20% | ~3 GB | - | ✅ FULLTEXT 索引有效 |
| 构建（mvn + npm） | ~80% | ~4 GB | ~2 GB 临时 | 🟡 构建期会短暂打满 |

**结论**：4 核 16 GB 配置可支撑 v1.1 全部计划需求，构建期注意错峰。

### 5.2 网络依赖风险

| 依赖 | 风险等级 | 说明 |
|------|---------|------|
| 智谱 GLM API | 🔴 高 | 完全依赖外网可达，API 中断则 Agent + 抽取 + 对比 + 时点全部失效 |
| Gitee | 🟡 中 | 仅构建/部署期需要 |
| Maven/npm 仓库 | 🟢 低 | 仅首次构建/更新依赖需要 |

**关键风险**：LLM 是单点故障。降级路径（Agent→老路径）可缓解 Agent 故障，但字段抽取、时点提取、语义对比等功能完全依赖 LLM。

### 5.3 智谱 API 限速约束

| 项 | 值 | 分析 |
|----|----|------|
| 限速 | 60 req/min | 全系统共享 |
| Agent 模式单次 | 2-5 次 LLM 调用 | 5 步循环最多 5 次 |
| 5 人同时 Agent | 10-25 次/分钟 | 接近限速上限 |
| 字段抽取（异步） | 1 次/文件 | 后台任务不阻塞用户 |

**建议**：如果并发用户 > 5 人，需要考虑升级智谱 API 套餐或增加调用缓存。

## 7. 架构建议

### 7.1 短期（v1.1 内）

| 建议 | 优先级 | 说明 |
|------|--------|------|
| 保留对智谱 API 的强依赖，不尝试本地 LLM | P0 | 硬件不支撑，网络可达前提下维持现状 |
| Agent SSE 流式降级为"全量返回" | P0 | 改 SSE 成本高，当前全量返回用户体验尚可 |
| 关键事实档案扩展架构 | P1 | 新增 4 表 + 事件流，复用现有 Entity/Repository/Service 模式 |
| 术语管理复用 dict 体系 | P1 | 新增一个 terminology type code，不用新建表体系 |

### 7.2 中长期

| 建议 | 说明 |
|------|------|
| 评估引入缓存层 | 大量重复查询（项目列表、字典）无缓存，可引入 Spring Cache + Caffeine |
| Agent 工具级降级 | 当前 Agent 级降级是 all-or-nothing，可细化到工具级异常处理 |
| 考虑数据灾备 | 单机 MySQL mysqldump 是唯一备份手段，可考虑 binlog 增量备份 |
| 监控告警 | 当前无 LLM 调用成功率/延迟监控，可补一个简单的定时检查脚本 |

### 7.3 不建议做 (Keep 清单)

```
❌ 流式 SSE 输出（成本 > 收益）
❌ 多供应商 LLM 动态切换（单机无场景）
❌ 本地离线 LLM 部署（硬件不达标）
❌ 微服务化（运维负担过重）
❌ Elasticsearch 引入（当前数据量级冗余）
❌ 细粒度字段级权限（当前 RBAC 够用）
```
