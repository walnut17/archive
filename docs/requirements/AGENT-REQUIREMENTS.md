# 智能问答 Agent 化 — 业务需求

> 文档说明: 本文档从**纯业务角度**梳理"用户期望智能问答能做什么",不涉及任何技术选型(LangChain4j / Spring AI / ReAct / 向量库等)。
> 版本: v1 / 2026-06-09
> 阅读对象: 产品 / 业务 / 测试 Agent
> 上游文档: `REQUIREMENTS-v1.md`(业务全貌)
> 下游用途: 架构 Agent 决定 agent 形态、测试 Agent 写验收用例

---

## 1. 用户角色与使用场景

### 1.1 角色(沿用 REQUIREMENTS-v1 §3)

| 角色 | 业务岗位 | 问答使用频率 |
|---|---|---|
| `admin` | 系统管理员 / 投委会办公室 | 中(验证能力、定位问题) |
| `user` | 投委会秘书 / 项目经理 / 业务部门 / 法务 | **高**(日常高频) |

> 智能问答对**两个角色能力完全一致**,本期不做角色差异化。

### 1.2 典型使用场景

- **A. 投委会秘书准备上会**:下周要审议 N 个项目,要快速回顾每个项目的关键风险 / 待落实问题。**当前痛点**: 只能搜出零散片段,无法把立项 / 申请 / 贷后串联。**期望**: 一次提问拿到项目**全周期**要点 + 引用。
- **B. 项目经理日常跟进**:手上 5-10 个项目并行,要问"走到哪一步 / 还欠什么材料 / 上次决议执行了没"。**当前痛点**: 状态散落在项目管理 / 待办 / 材料版本多个 tab。**期望**: 一句话问出"项目全貌",agent 跨表查。
- **C. 法务处理律师函**:收到律师函 → 上传 → 触发待办。法务想看"PRJ-001 今年几封 / 怎么处理的"。**当前痛点**: 无按项目聚合事件能力。**期望**: 直接出时间线。
- **D. 业务部门查金额**:"PRJ-001 现在剩余金额 / 已付出 / 已收回"。**当前痛点**: 现有问答**不查数据库**,答不上。**期望**: 数字问题必须查 MySQL 给准确值。
- **E. 投委会办公室应对外部问询**:"今年否决了几个项目 / 主要原因"。**当前痛点**: 无决议原文语义聚合。**期望**: 出列表 + 摘录。
- **F. 新员工快速学习**:问"投委会一般怎么审议固收类项目",基于历史报告给"一般规律"。
- **G. 管理员排查问答质量**:用户反馈"问 XX 答得不对",要回看提问历史、定位是检索还是 LLM 问题。**当前痛点**: 无问答历史、无过程留痕。**期望**: 每次问答有可追溯 trace。

## 2. 真实用户问题举例(带期望回答)

### 2.1 知识检索类(从文档原文找答案)

| # | 用户问题 | 期望回答要素 |
|---|---|---|
| 1 | "某项目的尽调报告主要风险点是什么?" | 风险点列表 + 引用 `立项报告 P.X 第 Y 段` |
| 2 | "去年所有收到律师函的项目是哪些?" | 项目清单(编号 + 名称)+ 材料链接 |
| 3 | "PRJ-001 的服务商是哪家?变更过几次?" | 服务商 + 历次变更时点(贷后议案链接) |
| 4 | "会议纪要里提到'风险预警'的有哪些?" | 纪要列表 + 引用段落 |
| 5 | "今年申请报告里被否决的最常见原因?" | 原因分类 + 频次(LLM 摘要聚合) |

### 2.2 数据库查询类(从 MySQL 取数,本期**新能力**)

| # | 用户问题 | 期望回答要素 |
|---|---|---|
| 6 | "PRJ-001 现在剩余金额多少?" | 精确数字(到分)+ 公式 `初始 - 累计付出 + 累计收回` |
| 7 | "今年结清的项目有几个?涉及总金额?" | 数字 + 项目清单 |
| 8 | "我有几条超期待办?分别是什么?" | 条数 + 列表(标题 / 截止日 / 关联项目) |
| 9 | "本月上传了多少份材料?哪个分类最多?" | 总数 + 分类分布 |
| 10 | "PRJ-002 的阶段走到哪一步了?" | 阶段 + 当前材料版本数 + 待办数 |

### 2.3 综合推理类(检索 + 查库 + 推理)

| # | 用户问题 | 期望回答要素 |
|---|---|---|
| 11 | "PRJ-001 立项时有 3 个待落实问题,申请时全回答了吗?" | 3 个问题 + 落实状态 ✓/✗ + 引用 |
| 12 | "这个待办为什么一直挂着?谁创建?超期多久?" | 详情 + 来源(规则 / 时点 / 手工)+ 超期天数 |
| 13 | "律师函规则是什么?执行情况?" | 规则定义 + 历次触发记录 |
| 14 | "如果收到法院传票,按规则会怎样?" | 规则匹配说明 + 应生待办(HIGH,3 天) |
| 15 | "PRJ-005 四阶段金额有没有前后矛盾?" | 四阶段金额对比表 |

### 2.4 典型"答得不好"抱怨

- "我问的是数字,你给我复制一段文字,我自己还得算"
- "你说'请查阅相关报告',我就是要你查啊"
- "前一轮说 PRJ-001,后一轮你忘了又问一遍"
- "你引用的 [3] 我点进去找不到"
- "你给的答案和报告里写的不一样,谁对?"
- "连问 3 次,系统都问'哪个项目',很烦"

## 3. 必须支持的能力(分类)

### 3.1 知识检索类(沿用现有 + 增强)

**必须**:MySQL FULLTEXT 全文检索(中文友好);按"项目 / 议案 / 材料 / 版本"过滤展示;**答案末尾必须标注引用来源** `[1] [2] [3]`(点击可跳原文);按相关性重排(LLM 或 MySQL 分数);**支持时间范围过滤**("今年/最近"自动提取)。

**应该**:多关键词加权(同义 / 近似词合并);跨材料聚合(同项目不同材料合并展示);输入时首字自动补全常见问法。

**可选**:错别字 / 同音字容错;用户可收藏常问问题。

### 3.2 数据库查询类(**新能力,本期重点**)

**必须**:
- 金额查询:`剩余 / 初始 / 累计付出 / 累计收回`,精确到分,返回时附带公式
- 阶段查询:某项目当前阶段 + 阶段历史
- 待办查询:超期 / 今日 / 本周,按项目 / 来源 / 优先级过滤
- 材料查询:某项目某阶段的材料份数、分类分布
- 项目列表:按阶段 / 起投时间 / 金额范围筛
- **自然语言直达 SQL**:`PRJ-001 多少钱` 自动转 `SELECT remaining_amount FROM project WHERE code='PRJ-001'`

**应该**:COUNT / SUM / GROUP BY 聚合;自然语言时间(今年 / 上月 / 过去 7 天)自动转 WHERE。

**明确不能**:
- ❌ 任何**写操作**(不能 INSERT / UPDATE / DELETE)
- ❌ 超过 3 张表 JOIN 的复杂查询
- ❌ 财务凭证明细(银行流水等,不在系统边界)

### 3.3 综合推理类(检索 + 查询 + LLM 思考)

**必须**:
- 多步推理:"查项目 ID → 查金额 → 查材料 → 检索报告"多步串起来
- 跨阶段关联:立项 / 申请 / 贷后 / 结清 4 阶段一次性拉通
- 规则 + 事实结合:既查"现在状态"又解释"为什么"(规则触发)
- **上下文记忆**:同一会话多轮对话,后续问题不丢前文实体(例:第一轮说"PRJ-001",后面"它多少钱"自动补全主语)

**应该**:数字必须以数据库实时值为准,LLM 不准"估算";LLM 推断时若非来自 DB,明确"根据 XX 报告推断,实际以数据库为准"。

**明确不能**:
- ❌ 预测性判断("未来 3 个月会违约吗" — 无数据支持,拒绝回答)
- ❌ 跨项目横向比较并给"投资建议"(只罗列事实)
- ❌ 议程 / 投票 / 决议结论(明确不在系统边界,REQUIREMENTS-v1 §1.2)

## 4. 用户体验期望

### 4.1 响应时间

| 问题类型 | 期望 | 容忍上限 |
|---|---|---|
| 简单知识检索 | < 2 秒 | 5 秒 |
| 数据库查询 | < 1 秒 | 3 秒 |
| 综合推理(多步) | < 5 秒 | 15 秒 |
| 长报告总结(50 页) | < 10 秒 | 30 秒 |

> **核心痛点**: 当前一次性 prompt 拼全文,常常超时 / token 超限。**期望:agent 拆步,避免单次超长上下文**。

### 4.2 可观测(用户能看到思考过程吗?)

**必须**:
- 每条结论附带"出处"(文档引用 / 数据库字段 / 推理步骤)
- 推理过程可折叠展示("查看思考过程")
- 检索 0 命中 / DB 超时 / LLM 报错时,前台给**具体错误原因**而不是"系统异常"

**应该**:agent 推理 > 3 秒时显示"正在第 X 步 / 共 N 步";展示"本次用了:全文检索 1 次、数据库查询 1 次、LLM 生成 1 次"。

**不要**:暴露 SQL 语句 / LLM 原始 prompt;强制用户每步"确认是否继续"(agent 自主跑完)。

### 4.3 失败降级

| 失败场景 | 用户期望 |
|---|---|
| 检索 0 命中 | "知识库中未找到相关内容,换个问法试试 / 用更具体的项目编号" |
| LLM 调用失败(智谱限流) | **仍返回检索结果 + 候选引用**,顶部提示"智能总结暂不可用,请直接查看下方材料" |
| 部分步骤失败 | 已成功的部分仍展示,不吞掉 |
| 网络超时 | "网络繁忙,请稍后重试"(不暴露 stacktrace) |

> **核心降级原则**: 任何环节失败,不能给"空答案"。能展示的就展示,不能展示的就给替代方案。

### 4.4 多轮对话

**必须**:同一会话内,后续问题自动带上"上文实体";用户随时"换个话题 / 重置上下文";历史问答可在"我的问答历史"翻看。

**应该**:显示"当前会话主题"标签。

### 4.5 界面前端(沿用现有 + 微调)

- 沿用:问答入口在 `Knowledge.vue`,不动菜单
- 调整:答案下加"引用:"可点跳材料版本详情页;长答案默认展开前 3 段后折叠;推理过程默认折叠("查看思考过程");多轮对话输入框下加历史 chip,点击恢复上文

### 4.6 本地 archive 只读工具 (v1.2 规划 · UP-0611-01)

- **需求**: [`REQUIREMENTS.md`](REQUIREMENTS.md) §5.6.7
- **Plan**: [`upgrade_to_settle/plan-2026-06-11-archive-local-fs-tools.md`](../../upgrade_to_settle/plan-2026-06-11-archive-local-fs-tools.md)
- **必须**: Agent 可在 `D:/archive/files` 与 `D:/archive/parsed` 内 **只读** list/grep/read；禁止越界 logs/config
- **应该**: 优先 `materialVersionId` 绑 DB 路径，再 grep/read；与 `search_fulltext` 互补

## 5. 风险与边界

### 5.1 业务边界(沿用 REQUIREMENTS-v1 §1.2)

| 业务 | agent 能否答 | 说明 |
|---|---|---|
| 项目档案 / 材料内容 / 金额 / 阶段 / 待办 / 时点 / 规则 | ✅ | 在范围内 |
| 议程编排 / 投票 / 决议原文 | ❌ | OA 管 |
| 财务凭证 / 银行流水 | ❌ | ERP 管 |
| 合同细节 | ❌ | 合同系统管 |
| 客户关系 / CRM | ❌ | CRM 管 |
| 移动端 | ❌ | 本期不做 |

> **agent 答不出的明确回复**: "该问题不在投委会档案系统管理范围内,请查阅 OA / ERP / 合同系统。"

### 5.2 数据准确性风险

| 风险 | 应对 |
|---|---|
| LLM 把"200 万"说成"20 万" | **数字必须以 DB 为准**,LLM 不准"算"只准"引" |
| 用户问的项目编号不存在 | **先校验项目编号**,不存在直接答"未找到" |
| 引用片段对不上原文(LLM 幻觉) | 引用必须是检索命中的真实 snippet,不允许 LLM 二次创作 |
| DB 查询性能(全表扫) | 强制走索引,慢查询告警 |

### 5.3 安全 / 权限

- **审计**:每次问答写到 `llm_call_log`(已有埋点),记录问题 / 步骤 / 工具调用 / 最终答案
- **越权**:不能通过"问"导出大量数据(例"把所有人名给我列出来"必须有项目 / 阶段过滤)
- **网络**:智谱 GLM 走内网代理,不向外网

### 5.4 体验风险

- **LLM 限流**: 免费但 60 req/min,agent 要会判断"该走全文检索就检索,不必每次问 LLM"
- **agent 死循环**:多步推理必须有步数上限(≤ 8 步),超时必须终止并返回当前结果
- **冷启动**:知识库为空时,问什么都答不出,要给"知识库暂无内容,先上传报告吧"

## 6. 验收场景(10 条)

> 每条对应"业务可实测",不依赖技术实现。

**1. 数据库查询(基础)** — 问"PRJ-001 现在剩余金额多少?";期望:精确数字 + 公式 + 时间戳;不通过:答"请查阅结清报告"。

**2. 多轮对话** — 第一轮"PRJ-001 是什么项目?",第二轮"它现在的阶段?";期望:第二轮自动带 PRJ-001,答"POST_LOAN,共 5 份材料";不通过:第二轮问"请提供项目编号"。

**3. 跨阶段推理** — "PRJ-005 立项时 3 个待落实问题,申请时全回答了吗?";期望:列 3 个问题 + 落实状态 + 引用;不通过:只返回片段或答"无法回答"。

**4. 规则解释** — "如果收到法院传票,按规则系统会怎样?";期望:解释规则(`应诉准备 HIGH 3 天`)+ 该项目已有待办;不通过:答"请咨询管理员"。

**5. 金额校验** — 问"PRJ-002 剩余金额",DB = `850,000.00`;期望:答 `¥850,000.00` 完全一致;不通过:答 `¥85 万`(精度错)或 `约 90 万`(估算)。

**6. 检索失败降级** — "外星人入侵怎么处理";期望:友好提示"未找到相关材料,试试更具体的项目编号";不通过:报 500 / 答"系统异常"。

**7. 长 context 不超限** — "总结 PRJ-001 立项报告所有内容"(50 页);期望:agent 拆步,分章节摘要后合并,5-10 秒返回;不通过:单次 prompt 塞 50 页,超时或 token 超限。

**8. 引用可点击** — "PRJ-001 服务商风险点";期望:末尾 `[1] 立项报告 P.12 第 3 段`,点击跳材料版本详情页并定位 P.12;不通过:引用是死的。

**9. 阶段 + 金额综合** — "列出所有贷后阶段、剩余金额 > 100 万的项目";期望:SQL 自动转,返回列表(代码 / 名称 / 阶段 / 剩余金额 / 材料版本数);不通过:让用户"用筛选器自己筛"。

**10. 审计可追溯** — 任意用户提问一次;期望:`llm_call_log` 有 1+ 条记录,scenario=QA,含问题 / 耗时 / 状态;涉及 DB 查询时能看到查了哪些表 / 字段;不通过:问答完成但无记录,无法排查。

## 7. 关键决策点(给后续 Agent)

> 业务没明说但 agent 化必须回答,留给架构 / 测试 Agent。

1. **项目编号格式**:`PRJ-XXX` 是示例还是真实?数字位数?
2. **多轮对话持久化**:会话存 Redis / DB / 客户端?
3. **问答历史保留期**:永久 / 90 天 / 30 天?
4. **跨用户隔离**:用户 A 的问题,用户 B 看到吗?(本期建议:不共享)
5. **并发限流**:同一用户 1 秒连点 5 次,队列 / 合并 / 拒绝?
6. **智谱限流**:LLM 60 req/min 全局,agent 化后调用量上升,需不需要排队?

## 8. 不在范围内(明确排除,本期不做)

- ❌ 主动推送(agent 主动发"你可能想问")
- ❌ 跨项目"投委会总体审议偏好"分析(粒度太粗,数据不够)
- ❌ 问答与待办双向联动("答完问题自动标完成")
- ❌ 多语言(只中文)
- ❌ 移动端语音输入
- ❌ 图表 / 可视化自动生成(答文字即可)
